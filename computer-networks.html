What is the Internet?

The Internet is a global network of interconnected computing devices that communicate using standardized protocols. It can be understood from two complementary perspectives:

🔧 "Nuts and Bolts" View

Hardware & software components: hosts, communication links (fiber, copper, radio), packet switches (routers & link-layer switches), ISPs, protocols (TCP, IP, HTTP, etc.).

🧩 Service View

Infrastructure that provides services to distributed applications (Web, email, gaming, streaming). Offers a socket interface — a set of rules for how apps request/receive data.

Key Components

  • Hosts / End Systems — devices that run network applications (laptops, phones, servers).
  • Communication Links — physical media: fiber optic, coaxial cable, copper wire, radio spectrum. Characterized by transmission rate (bandwidth) in bits/second (bps).
  • Packet Switches — routers (network layer) and link-layer switches (data-link layer) forward packets toward their destination.
  • ISPs (Internet Service Providers) — hierarchical network of networks through which end systems connect to the Internet.
  • Protocols — define the format and order of messages sent/received and the actions taken on message transmission or receipt.
  • Internet Standards — developed by IETF (Internet Engineering Task Force). Documents called RFCs (Request for Comments).
📌 Key Definition

Protocol: A protocol defines the format and the order of messages exchanged between two or more communicating entities, as well as the actions taken on the transmission and/or receipt of a message or other event.

Physical Media

🔌 Guided Media

Signals travel over solid medium: twisted-pair copper wire (most common), coaxial cable (HFC networks), fiber optic cable (immune to EM interference, very high speed).

Signals propagate freely: terrestrial radio (WiFi, LTE), satellite radio (geostationary ~35,000 km, LEO ~500–1,500 km). No physical wire needed.

03

Network Edge & Core

🖥 Network Edge

End systems (hosts) — clients and servers that run application programs. Located at the edge of the Internet. Examples: PCs, smartphones, web servers, IoT devices.

🌐 Network Core

The mesh of routers and links interconnecting all end systems. Core does not run user applications — it solely forwards packets toward destinations.

Two Approaches to Moving Data Through the Core

📦 Packet Switching

Messages broken into packets. Each packet is forwarded independently. No resource reservation — uses store-and-forward transmission. Used by the Internet.

📞 Circuit Switching

End-to-end resources reserved for the duration of a call (like a telephone call). Guaranteed bandwidth but wasteful when idle. Uses FDM or TDM multiplexing.

Store-and-Forward Transmission

In packet switching, the router must receive the entire packet before it can begin transmitting it on the outbound link.

Delay to send 1 packet of L bits over N links at rate R bps:
d = N × (L / R)
/* Each hop: wait for full packet, then forward */

FDM vs TDM (Circuit Switching)

📶 FDM — Frequency Division Multiplexing

The frequency spectrum of a link is divided into frequency bands. Each connection gets its own band continuously.

⏱ TDM — Time Division Multiplexing

Time is divided into frames, further divided into slots. Each circuit gets a dedicated time slot in every frame.

💡 Packet Switching Wins

Packet switching allows more users to share the network (statistical multiplexing). Example: 1 Mbps link, 10 users each needing 100 kbps when active but active only 10% of the time → packet switching can support ~35 users with the same probability of blocking as circuit switching supports 10.

✦ Exam Questions
Q1
A message of 7.5 Mbits is sent over a path with 3 links, each 1.5 Mbps. Ignoring propagation delay, how long does it take using store-and-forward packet switching?
Hint: d = N × (L/R) = 3 × (7.5M/1.5M) = 3 × 5 = 15 seconds
Q2
Why is packet switching considered more efficient than circuit switching for bursty data traffic?
Hint: Statistical multiplexing, no reserved but idle bandwidth.

Traffic Intensity & Queueing Delay

Let a = average packet arrival rate (packets/sec), L = packet length (bits), R = link rate (bps).

Traffic Intensity I = (L × a) / R

• I ≈ 0 → d_queue small
• I → 1 → d_queue grows very large
• I > 1 → packets arrive faster than served → infinite queue / loss
⚠ Rule: Never Design for I ≥ 1

When traffic intensity ≥ 1, the queue grows without bound. Average queueing delay → ∞. Always keep traffic intensity well below 1.

Packet Loss

Each router has a finite buffer (queue). When a packet arrives and the buffer is full, the packet is dropped (lost). The fraction of lost packets is the packet loss rate.

  • Lost packets may be retransmitted by the sender (TCP) or not at all (UDP).
  • Packet loss increases as traffic intensity approaches and exceeds 1.

Throughput

Throughput is the rate at which bits are successfully transferred from source to destination.

Instantaneous Throughput

Rate at a specific instant in time (bits/sec).

Average Throughput

Total bits transferred ÷ total time. F / T bits/sec.

Bottleneck Link

Throughput is limited by the bottleneck link — the link with the minimum capacity along the path.

Throughput = min(R_s, R_c, ..., R_n)
/* R_s = server rate, R_c = client rate, intermediate links */
📌 Example

A file of F = 32×10⁶ bits is transferred. Server link R_s = 2 Mbps, Client link R_c = 1 Mbps. Throughput = min(2, 1) = 1 Mbps. Transfer time = F / Throughput = 32s.

✦ Exam Questions
Q1
Differentiate between transmission delay and propagation delay with an example. Can propagation delay be reduced by increasing link bandwidth?
Hint: Transmission = pushing bits onto wire (depends on R). Propagation = bits travelling through wire (depends on physical distance). Increasing R only reduces transmission delay.
Q2
If traffic intensity I = 0.9, what happens to queueing delay? What if I = 1.0?
Hint: I=0.9 → large but finite delay. I≥1.0 → queue grows unbounded → loss.
Q3
A 1000-byte packet travels over a 10 Mbps link. Calculate the transmission delay. If the link is 500 km long (propagation speed 2×10⁸ m/s), what is the propagation delay?
Hint: d_trans = (1000×8) / 10M = 0.8 ms. d_prop = 500,000 / 2×10⁸ = 2.5 ms.

Layer-by-Layer Description (TCP/IP)

🔵 Layer 5 — Application Layer

Where network applications and their protocols live. Protocols: HTTP (web), SMTP (email), FTP (file transfer), DNS (name resolution). PDU = Message.

🟣 Layer 4 — Transport Layer

Transports application-layer messages between application endpoints. Protocols: TCP (reliable, connection-oriented), UDP (unreliable, connectionless). PDU = Segment.

🟠 Layer 3 — Network Layer

Routes datagrams from source to destination host. Protocol: IP (Internet Protocol). Also includes routing protocols (OSPF, BGP). PDU = Datagram.

🟢 Layer 2 — Data Link Layer

Moves frames between adjacent network nodes over a single link. Protocols: Ethernet, WiFi (802.11), PPP. PDU = Frame.

🟡 Layer 1 — Physical Layer

Moves individual bits within the frame from one node to the next. Depends on physical medium (copper, fiber, radio). PDU = Bit.

⚠ TCP/IP vs OSI

The TCP/IP model combines OSI layers 5, 6, 7 into the Application layer. OSI layers 1–4 correspond directly to TCP/IP layers 1–4. The Internet uses TCP/IP in practice; OSI is mainly a reference model.

Historical Perspective

Understanding the history of computer networks helps explain why the Internet is designed the way it is. The Internet's development spans over 60 years.

1961 – 1972
Packet Switching is Born
Kleinrock (1961) proved packet switching would work. Baran (RAND, 1964) proposed it for military use. Roberts (1967) proposed ARPANET. ARPANET went live in 1969 with 4 nodes (UCLA, Stanford, UCSB, Utah). First host-to-host protocol: NCP.
1972 – 1980
Internetworking & TCP/IP
ARPANET grows to ~200 nodes. Cerf and Kahn (1974) propose TCP/IP — the foundational protocol still used today. Ethernet (1976), ALOHAnet (satellite), Telenet (first public packet network).
1980 – 1990
Proliferation of Networks & DNS
ARPANET transitions to TCP/IP (Jan 1, 1983). DNS introduced (1983). NSFNET created (1986) to connect US universities at 56 kbps → later 1.5 Mbps → 45 Mbps. Dramatic growth to 100,000+ hosts.
1990 – 2000
The Web Explodes
Tim Berners-Lee invents the WWW (1991) at CERN. First web browser Mosaic (1993). ARPANET decommissioned (1991). ISPs emerge. Dot-com boom. Internet users grow from 1 million (1992) to 50 million (1997) to 100 million (1999). Killer apps: email, WWW, instant messaging.
2000 – 2010
Broadband & New Applications
Broadband home access (DSL, cable). P2P file sharing (Napster, BitTorrent). VoIP (Skype). Social networking (Facebook, YouTube). WiFi proliferation. ~1 billion users by 2005. Google becomes dominant.
2010 – Present
Mobile, Cloud & IoT Era
Smartphones dominate Internet access. Cloud computing (AWS, Azure). Video streaming consumes majority of traffic (Netflix, YouTube). 4G/5G wireless. IoT devices. ~5 billion users today. AI/ML reshaping network management.

Principles of Network Applications

A network application consists of programs that run on different end systems and communicate with each other over a network. Network apps live only at end systems — no software runs on network-core devices (routers/switches).

Application Architectures

🖥 Client-Server Architecture

Server: always-on host, permanent IP, often in data centers.
Client: communicates with server, may have dynamic IP, does not directly communicate with other clients.
Examples: Web (HTTP), Email (SMTP), FTP.

🔗 Peer-to-Peer (P2P) Architecture

No always-on server. Arbitrary end systems (peers) communicate directly. Peers are intermittently connected and change IP addresses. Highly scalable but hard to manage.
Examples: BitTorrent, Skype (legacy).

Processes Communicating

Communication actually occurs between processes (programs running on hosts), not between programs themselves. A process sends/receives messages through a socket.

Transport Services Required by Apps

📦 Data Integrity / Reliability

Some apps (file transfer, web transactions) require 100% reliable data transfer. Others (audio, video) can tolerate some loss.

⚡ Timing / Low Latency

Real-time apps (internet telephony, gaming) require low delay. Non-real-time apps (email, file transfer) are delay-tolerant.

📶 Throughput

Multimedia apps need a minimum throughput guarantee. Elastic apps (email, web) make use of whatever is available.

🔒 Security

Encryption, data integrity, end-point authentication. SSL/TLS provides these at the application layer on top of TCP.

M2·02

The Web and HTTP Protocol

The World Wide Web is an application-layer distributed information service. The HyperText Transfer Protocol (HTTP) is its application-layer protocol — the backbone of data communication on the Web.

📄 Web Page

Consists of a base HTML file and several referenced objects (images, CSS, JavaScript). Each object is addressable by a URL.

🔗 URL Structure

http://www.example.com/path/file.html
protocol://hostname/pathname

HTTP Overview

  • HTTP uses a client-server model — browsers are clients, web servers are servers.
  • HTTP is built on top of TCP (port 80). Client initiates TCP connection; server accepts; HTTP messages exchanged; TCP connection closed.
  • HTTP is stateless — server maintains no information about past client requests. (Cookies used to add state.)

Non-Persistent vs Persistent HTTP

🔁 Non-Persistent HTTP (HTTP/1.0)

A new TCP connection is opened for each object. After sending the object, the server closes the connection.

RTT: 2 RTTs + file transmission time per object.

♻️ Persistent HTTP (HTTP/1.1+)

Multiple objects can be sent over a single TCP connection. Server leaves connection open after sending response. Pipelining: client can send multiple requests without waiting for responses.

RTT: 1 RTT for all subsequent objects.

Non-Persistent HTTP Response Time = 2RTT + File Transmission Time
/* 1 RTT for TCP handshake + 1 RTT for HTTP request/response */

Cookies — Adding State to HTTP

Since HTTP is stateless, cookies allow websites to track users across requests. Components:

  • Cookie header line in HTTP response: Set-cookie: 1678
  • Cookie header line in next HTTP request: Cookie: 1678
  • Cookie file on client's browser/host.
  • Back-end database at Web server.

Web Caches (Proxy Servers)

A web cache (proxy server) satisfies HTTP requests on behalf of an origin server. The browser sends all HTTP requests to the cache. If the cache has the object, it returns it; otherwise it fetches from origin server.

💡 Why Web Caching?

Reduces response time for client requests. Reduces traffic on institution's access link. Enables poor content providers to deliver content effectively.

Conditional GET

Cache sends request with If-modified-since: <date>. If object not modified since date, server returns 304 Not Modified (no body) — saves bandwidth.

HTTP/2 and HTTP/3

HTTP/2 (2015)

Multiplexing multiple requests over single TCP connection. Server push. Header compression (HPACK). Eliminates head-of-line blocking within a connection.

HTTP/3 (2022)

Built on QUIC (UDP-based). Eliminates TCP head-of-line blocking entirely. Faster connection setup. Better for mobile (handles IP changes).

  • Uses TCP port 25. Transfers messages from client to server and between servers.

Mail Access Protocols

📬 POP3 (Post Office Protocol v3)

Downloads mail from server to local client. Three phases: authorization, transaction, update. Two modes: download-and-delete or download-and-keep. Stateless across sessions. Port 110.

📮 IMAP (Internet Mail Access Protocol)

Keeps all messages on server. Allows user to organize messages in folders on server. More complex state maintained across sessions. Port 143. Preferred for multi-device access.

📌 SMTP vs HTTP Comparison

Both use TCP, transfer files (objects). HTTP: pull protocol — client pulls from server. SMTP: push protocol — sending server pushes to receiving server. HTTP: each object in its own response message. SMTP: multiple objects in multipart message.

The Internet Directory Service — DNS

The Domain Name System (DNS) is the Internet's directory service. It translates human-readable hostnames (www.google.com) into machine-readable IP addresses (142.250.80.46). DNS is a distributed database implemented in a hierarchy of name servers, and an application-layer protocol using UDP port 53.

Services Provided by DNS

  • Hostname-to-IP translation — primary purpose.
  • Host aliasing — canonical hostname + alias names (e.g., relay1.west-coast.enterprise.com → enterprise.com).
  • Mail server aliasing — MX records map mail domains to mail servers.
  • Load distribution — replicated servers: set of IP addresses for one canonical name; DNS rotates order of addresses.

Peer-to-Peer Applications

In a P2P architecture, pairs of intermittently connected hosts (peers) communicate directly. There is no always-on server. Peers contribute resources (bandwidth, storage) as well as consume them — this gives P2P its hallmark self-scalability.

P2P File Distribution — BitTorrent

BitTorrent is the dominant P2P file distribution protocol. Key concepts:

  • Torrent — a group of peers exchanging chunks of a file.
  • Tracker — a server that tracks the peers participating in a torrent (provides peer list).
  • Chunks — files divided into 256 KB chunks. Peers download and upload chunks simultaneously.
  • Rarest First — peer requests the rarest chunks first (those fewest neighbours have) to promote availability.
  • Tit-for-Tat — peers prefer to send chunks to neighbours who are sending chunks back at the highest rate (4 unchoked + 1 optimistically unchoked).
📥 Leecher

A peer that is downloading the file but has not yet completed it. Contributes partial chunks it has.

🌱 Seeder

A peer that has the complete file and continues uploading to others. Essential for torrent health.

P2P vs Client-Server File Distribution Time

For distributing a file of size F to N peers, with server upload capacity u_s, peer upload capacities u_i, and min download capacity d_min:

Client-Server Distribution Time:
T_cs = max{ NF/u_s , F/d_min }

P2P Distribution Time:
T_p2p = max{ F/u_s , F/d_min , NF/(u_s + Σu_i) }
/* P2P: total upload capacity grows with N peers → self-scaling */
💡 Self-Scalability Insight

In client-server, distribution time grows linearly with N (server bottleneck). In P2P, as N grows, so does total upload capacity — distribution time grows very slowly. This is why BitTorrent can distribute to millions of users efficiently.

Distributed Hash Tables (DHT)

A DHT is a distributed P2P database used to store and retrieve (key, value) pairs. Each peer holds a small subset of the database. Examples: Chord, Kademlia (used by BitTorrent's DHT tracker-less mode).

  • Keys are hashed to integers in [0, 2ⁿ). Each peer is responsible for a range of keys.
  • Provides O(log N) routing — queries are routed through O(log N) peers to find a key.
  • Resilient to churn — peers constantly joining/leaving.

M3·01

Introduction to the Transport Layer

The transport layer provides logical communication between application processes running on different hosts. While the network layer provides logical communication between hosts, the transport layer provides logical communication between processes.

🏠 On the Sending Side

Transport layer takes application messages, breaks them into segments, adds a transport-layer header, and passes the segment to the network layer.

📬 On the Receiving Side

Transport layer receives segments from network layer, reassembles them into messages, and delivers to the correct application process via its socket.

📌 Transport vs Network Layer

Network layer (IP): Logical communication between hosts — best-effort delivery, unreliable, no guarantees.
Transport layer (TCP/UDP): Logical communication between processes — TCP adds reliability, ordering, congestion control on top of IP.

Internet Transport Protocols

TCP — Transmission Control Protocol

Reliable, in-order delivery. Congestion control. Flow control. Connection setup. Used by: HTTP, SMTP, FTP, SSH.

UDP — User Datagram Protocol

Unreliable, unordered delivery. No frills, no overhead. Used by: DNS, streaming media, VoIP, online gaming.

⚠ What transport layer CANNOT do

Transport layer cannot control network-core delay, bandwidth, or loss — these are properties of the network layer. TCP can only react to congestion, not eliminate it.

Connectionless Transport: UDP

UDP (User Datagram Protocol) — RFC 768 — is a bare-bones, no-frills transport protocol. It is connectionless: no handshaking, no state maintained. Each UDP segment is handled independently.

Why Use UDP?

  • No connection establishment → no RTT delay added before sending.
  • No connection state → server can support more active clients.
  • Small header overhead → only 8 bytes vs TCP's 20 bytes.
  • No congestion control → can blast away at any rate desired.
  • Application controls what data is sent and when (important for real-time).

Sender: treat segment contents as 16-bit integers. Compute 1s complement sum of all integers. Put result in checksum field. Receiver: compute 1s complement sum of all received integers (including checksum). If result is all 1s → no error detected. Otherwise → error.

UDP checksum detects errors but does NOT correct them. If an error is detected, the segment is discarded. It also cannot detect all errors — complementary errors in two words may cancel out.

Sender allowed to have multiple unacknowledged packets "in flight." Two approaches:

Receiver: only ACKs correctly-received in-order packet — cumulative ACK.

Receiver buffer: Only 1 packet needed (discards out-of-order).

Sender window: up to N consecutive unACKed packets.
Receiver: individually ACKs each correctly-received packet (even out-of-order).
On timeout: retransmit only the specific lost packet.
Receiver buffer: Buffers out-of-order packets (window size N).

M3·07

Principles of Congestion Control

Congestion occurs when too many sources are sending too much data too fast for the network to handle. Manifests as: long queueing delays, packet loss (buffer overflow at routers).

⚠ Costs of Congestion

1. Large queueing delays as packet arrival rate → link capacity.
2. Retransmissions needed for dropped packets → wasted bandwidth.
3. Unnecessary retransmissions due to large delays → further waste.
4. When packet dropped, all upstream transmission capacity used on that packet was wasted.

Three Scenarios of Congestion

Scenario 1: Infinite Buffers

No packet loss. But as arrival rate → link capacity R, delay → ∞. Throughput saturates at R/2 for 2 senders.

Scenario 2: Finite Buffers + Retransmit

Packets lost when buffers full. Sender retransmits. Goodput < offered load because some transmitted packets are retransmissions of lost packets.

Approaches to Congestion Control

End-to-End Congestion Control

No explicit feedback from network. Congestion inferred from packet loss and delay. Approach taken by TCP. Network layer doesn't help.

Network-Assisted Congestion Control

Routers provide explicit feedback to senders. E.g., a single bit set in packet header (ATM, DECbit). Modern approach: ECN (Explicit Congestion Notification) — 2 bits in IP header.

📌 ECN — Explicit Congestion Notification

Router marks packets (ECN bit) when experiencing congestion instead of dropping them. Receiver echoes congestion indication to sender via TCP header. Sender reduces rate. Avoids packet loss while still signaling congestion.

TCP Congestion Control

TCP uses an AIMD (Additive Increase, Multiplicative Decrease) approach. The sender maintains a congestion window (cwnd) that limits the amount of unACKed data in the network.

Effective Sending Rate ≈ cwnd / RTT bytes/sec
Max in-flight data: min(cwnd, rwnd)
/* cwnd = congestion window, rwnd = receiver window */

TCP Congestion Control Algorithm — Three Phases

Slow Start
cwnd grows exponentially
cwnd × 2 per RTT
Congestion Avoidance
cwnd grows linearly (AIMD)
cwnd + MSS²/cwnd per RTT

Slow Start

  • Initial cwnd = 1 MSS (Maximum Segment Size). ssthresh = large value initially.
  • On each ACK received: cwnd += 1 MSS → cwnd doubles every RTT (exponential).
  • Ends when cwnd reaches ssthresh → enters congestion avoidance.
  • Also ends on loss event (timeout or 3 dup ACKs).

Congestion Avoidance (AIMD)

  • cwnd += MSS × (MSS/cwnd) per ACK → linear increase of 1 MSS per RTT.
  • On timeout loss: ssthresh = cwnd/2; cwnd = 1 MSS; restart slow start.
  • On 3 duplicate ACKs: ssthresh = cwnd/2; cwnd = ssthresh + 3 MSS → enter Fast Recovery.

Fast Recovery

  • Triggered by 3 duplicate ACKs (less severe than timeout — some packets still flowing).
  • cwnd = ssthresh + 3 MSS. For each additional dup ACK: cwnd += 1 MSS.
  • On new ACK (non-duplicate): cwnd = ssthresh → enter Congestion Avoidance.
  • Recommended (not mandatory): TCP Reno uses fast recovery; TCP Tahoe does not.

Introduction to the Network Layer

The network layer is responsible for moving packets from a sending host to a receiving host across multiple networks. Every router in the Internet implements the network layer.

Two Key Network-Layer Functions

📍 Routing

Determining the route (path) taken by packets from source to destination. Done by routing algorithms that populate the router's forwarding table. A control-plane function.

⚡ Forwarding

Moving a packet from a router's input link to the appropriate output link. Done locally using the forwarding table. A data-plane function. Must be extremely fast (nanoseconds).

M4·02

Virtual Circuits & Datagram Networks

Network-layer packet delivery can follow two paradigms: virtual-circuit (VC) networks (connection-oriented) and datagram networks (connectionless). The Internet uses the datagram model.

🔗 Virtual Circuit Networks

A path (virtual circuit) is established before data transfer. Each packet carries a VC identifier, not a full destination address. Routers maintain per-connection state. Resources (bandwidth, buffers) may be reserved. Examples: ATM, Frame Relay, MPLS.

📦 Datagram Networks

No connection setup. Each packet carries a full destination IP address. Each router makes its own forwarding decision independently. No per-connection state at routers. Examples: Internet (IP). Simpler router design; harder to guarantee QoS.

Virtual Circuit — Three Phases

  1. VC Setup — call initiation: path chosen, VC IDs assigned, resources reserved, forwarding tables updated at each router.
  2. Data Transfer — packets flow along the established path using VC IDs for forwarding.
  3. VC Teardown — resources freed, forwarding table entries removed at each router.

Switching Fabrics

Memory

Traditional: CPU copies packet from input to memory to output. Speed limited by memory bandwidth. Max: B/2 bps (B = memory bandwidth).

Bus

Input port writes packet to shared bus; all output ports receive it; only correct port keeps it. Bandwidth limited by bus speed. Collision on bus.

Crossbar

N×N switch fabric. Allows N simultaneous transfers if different input and output ports. Most scalable. Used in modern high-speed routers.

Output Port Queueing & Scheduling

  • HOL (Head-of-Line) Blocking: A queued packet at input port blocks other packets behind it even if their output port is free — occurs with input queueing.
  • FIFO Scheduling: First-in, first-out. Simple but no QoS.
  • Priority Queuing: Multiple priority classes. Always serve highest priority first.
  • Round Robin (RR): Cyclically serve each class. Weighted Round Robin (WRR) gives different weights to classes.
  • WFQ (Weighted Fair Queuing): Generalisation of RR — each class gets weighted min-max fair share of bandwidth.

NAT — Network Address Translation

NAT allows an entire private network to share a single public IP address. The NAT router replaces private IP:port with the public IP:new port in outgoing packets, and reverses the translation for incoming packets using a NAT translation table.

✅ NAT Benefits

Conserves IPv4 addresses. Hides internal network structure. Security through obscurity — external hosts cannot directly initiate connections to internal hosts.

❌ NAT Controversies

Violates end-to-end principle. Breaks P2P applications (NAT traversal needed). Routers should only process up to L3 — NAT modifies port numbers (L4). IPv6 is the proper solution.

IPv6

  • 128-bit addresses — 340 undecillion addresses. No exhaustion problem.
  • Fixed 40-byte header — simplified processing (no checksum, no fragmentation at routers).
  • No fragmentation — source performs path MTU discovery.
  • No checksum — reliability left to transport layer.
  • Flow label — identifies flows for special QoS treatment.
  • Tunneling: IPv6 datagrams carried as payload in IPv4 datagrams to traverse IPv4-only routers.

M4·05

Routing Algorithms

A routing algorithm determines the least-cost path between a source and destination through the network. The network is modeled as a graph G(N,E) where N = set of routers, E = set of links with costs.

Classification of Routing Algorithms

Global (Link-State)

All routers have complete knowledge of the network topology and link costs. Uses Dijkstra's algorithm. Requires link-state broadcasts. Example: OSPF.

Decentralised (Distance-Vector)

Each router knows only its directly connected neighbours and their link costs. Exchanges cost vectors with neighbours iteratively. Uses Bellman-Ford equation. Example: RIP, BGP.

Link-State Algorithm — Dijkstra's

Computes the least-cost path from one source to all other nodes. After k iterations, knows the least-cost path to k destinations. Complexity: O(N² ) basic; O(N log N) with priority queue.

Notation:
D(v) = current least-cost path estimate from source to v
p(v) = predecessor node along least-cost path to v
N' = set of nodes whose least-cost path is definitively known

Initialization: D(u) = c(src,u) for all neighbours u; D(v) = ∞ otherwise
Loop: select w ∉ N' with min D(w) → add to N'
update D(v) = min(D(v), D(w)+c(w,v)) for each neighbour v of w

Distance-Vector Algorithm — Bellman-Ford

Each node x maintains a distance vector D_x = [D_x(y) : y ∈ N]. Periodically sends its DV to neighbours. On receiving a neighbour's DV, updates its own using:

D_x(y) = min_v { c(x,v) + D_v(y) } for each y ∈ N
/* x's estimate to y = best of: cost to neighbour v + v's estimate to y */
  • Decentralised — no global view needed.
  • Self-terminating — updates stop when no DV changes.
  • Count-to-infinity problem: bad news (link failure) propagates slowly. Poisoned reverse partially solves this.
  • Poisoned Reverse: If Z routes through Y to X, Z tells Y its distance to X is ∞ (lie) — prevents Y from routing through Z to reach X.

M4·06

Routing in the Internet

The Internet is organized as a collection of Autonomous Systems (ASes). Each AS is a group of routers under the same administrative control (e.g., a company's network, an ISP). This gives rise to two tiers of routing:

🏠 Intra-AS Routing

Routing within an AS. All routers in an AS run the same intra-AS protocol. Also called Interior Gateway Protocol (IGP). Goal: performance (least cost). Examples: RIP, OSPF.

🌍 Inter-AS Routing

Routing between ASes. Policy-driven (business relationships). A single protocol used across the entire Internet: BGP (Border Gateway Protocol). Gateway routers run both intra-AS and inter-AS protocols.

RIP — Routing Information Protocol

  • Distance-vector algorithm. Uses hop count as metric (not bandwidth). Max 15 hops (16 = infinity).
  • Routers exchange RIP advertisements (DVs) every 30 seconds via UDP port 520.
  • If no advertisement from a neighbour in 180 seconds → link assumed dead.
  • Simple but limited scalability. Used in small networks.

OSPF — Open Shortest Path First

  • Link-state algorithm. Uses Dijkstra's algorithm locally. Weights can be bandwidth, delay, etc.
  • Link-state advertisements (LSAs) flooded to entire AS (not just neighbours). Carried directly over IP (protocol number 89).
  • Advantages over RIP: security (authenticated messages), supports multiple same-cost paths, hierarchical OSPF (backbone + local areas).
  • Hierarchical OSPF: Areas with area border routers connected to backbone (area 0).

BGP — Border Gateway Protocol

BGP is the "glue" of the Internet — the inter-AS routing protocol that makes global routing possible. It is a path-vector protocol (variant of DV), advertising complete AS paths to avoid loops.

  • eBGP (external BGP): Between border routers of different ASes. Obtains subnet reachability from neighbouring ASes.
  • iBGP (internal BGP): Between routers within the same AS. Distributes externally-learned routes within the AS.
  • BGP Session: TCP connection (port 179) between BGP peers (semi-permanent).
  • BGP Attributes: AS-PATH (list of ASes a route has traversed), NEXT-HOP (IP of the border router to the next AS).
  • Route Selection: Policy → shortest AS-PATH → closest NEXT-HOP (hot-potato routing) → BGP ID.
📌 Hot Potato Routing

When an AS has multiple exit points to reach a destination, it forwards packets to the exit point with the least cost within the AS (closest gateway) — like dropping a "hot potato" as quickly as possible, regardless of what happens outside the AS.

Broadcast & Multicast Routing

Beyond unicast (one-to-one), networks support broadcast (one-to-all) and multicast (one-to-many-specific) delivery.

Broadcast Routing

Deliver a packet from a source to all other nodes in the network. Key challenge: avoid duplicate deliveries and loops.

Reverse Path Forwarding (RPF)

When a router receives a broadcast packet from source S:

  • If packet arrived on the link that is on the shortest path back to S → forward on all other links.
  • If packet arrived on any other link → discard (likely a duplicate).
💡 RPF Advantage

RPF leverages existing unicast routing tables — no extra state needed. Efficient and simple. Used as a building block for multicast routing (PIM-DM).

Multicast Routing

Deliver packets from a source to a subset of nodes (multicast group). Receivers express interest by joining a multicast group (identified by a Class D IP address: 224.0.0.0/4).

Internet Group Management Protocol (IGMP)

  • Used between a host and its directly-attached router (not between routers).
  • Host sends IGMP join message to indicate it wants to receive a multicast group's traffic.
  • Router periodically sends IGMP queries; hosts respond to confirm membership.
  • If no host responds to a group query → router stops forwarding that group's traffic on that link.

Multicast Routing Algorithms

Source-Based Tree

Separate shortest-path tree per source. Minimum delay to all group members from that source. Used by PIM-DM (Protocol Independent Multicast — Dense Mode). Scales poorly with many sources.

Shared Tree (Group-Shared)

Single shared tree rooted at a Rendezvous Point (RP) for the entire group. Sources send to RP; RP distributes. Used by PIM-SM (Sparse Mode). More scalable; single RP = bottleneck/failure point.
